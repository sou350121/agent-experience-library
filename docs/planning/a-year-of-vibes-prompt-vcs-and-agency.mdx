# 《A Year Of Vibes》读后：Prompt 版本控制、协作界面与“责任归属”

> Armin Ronacher 在《A Year Of Vibes》里描述了一个典型的 2025：
>
> **写代码的人变少了，指挥 AI 的人变多了；工程经验还在，但决策越来越依赖“vibes”。**

本文基于：
- 原文：`https://lucumr.pocoo.org/2025/12/22/a-year-of-vibes/`
- 宝玉（`@dotey`）二次总结与观察

## 1. “Vibes”到底是什么？

在 AI 编程场景里，很多选择短期无法被量化：
- 你更愿意用哪个工具？（Amp / Claude Code / Codex / Copilot…）
- 你更信任哪种交互？（CLI vs IDE vs Web）
- 你更能忍受哪种失败形态？（慢但稳 vs 快但易出坑）

**Vibes** 指的是这种“难以形式化、只能凭直觉感受”的选择机制。

### 工程化建议：把 vibes 变成“可比较的主观量表”
你可以给每个工具维护一张小卡片（每月更新一次）：
- **上手摩擦**：配置/切换成本高不高？
- **可控性**：偏航时能否快速拉回？
- **失败代价**：错一次要花多久补救？
- **信任建立**：我敢不敢让它独立跑 30 分钟？

## 2. 人与工具的关系：parasocial bond（单向亲密感）

当工具具备“记忆”与“人格影子”，开发者很容易产生单向亲密感。

### 关键边界：agency 与责任
- **agent** 这个词容易让人误以为系统拥有“自主性与责任”。
- 但在工程与治理层面，**责任应该永远留在人类手里**。

工程化建议：在你的团队约定里写清：
- 谁对上线负责？谁对安全负责？谁对数据合规负责？
- AI 只能是执行者/建议者，不是责任主体。

## 3. 最大的工具欠账：Prompt 版本控制（Prompt VCS）

Armin 的痛点非常真实：
- 代码是 AI 生成时，**光看 diff 无法判断质量**。
- 你需要知道：
  - 触发这段代码的 prompt 是什么？
  - 中间走过哪些失败路径？
  - 为什么最后选择了这个实现？

### 可落地的最小方案（建议直接用在你的仓库里）

把“提示词与对话”当成一等公民，跟代码一起入库：

```text
stories/
  S-0007-xxx.md
sessions/
  S-0007/
    2025-12-23-codex.md
    2025-12-23-claude-code.md
    failures.md
prompts/
  S-0007.md
```

#### `prompts/S-0007.md` 推荐字段
- 目标/边界/验收
- 关键上下文（链接到 docs 或文件路径）
- 使用的模型与模式（如 reasoning effort）
- 主提示词（可复制）
- 反例提示词（哪些说法会让它偏航）

#### `sessions/S-0007/failures.md` 的价值
失败路径对 AI 来说是“宝贵的”：
- 你把 AI 拉回早期状态时，希望它**记得哪些路走不通**。
- 这份失败记录能显著减少“重蹈覆辙”的概率。

> 与本文库的连接：可以把这些文件挂到 `docs/features/S-0007/` 的状态账本里（见 `planning/agentic-coding-doc-engineering`）。

## 4. PR 协作界面不适配：你无法“正式 review 自己的 PR”

在 AI 编程里，开发者常见动作是：
- 在 PR 里给 AI 下指令（让它补测试/改结构/修边界）
- 自己充当 reviewer，但需要留下结构化指令与证据

### 可落地做法：把“自审”标准化
在 PR description 里加一个区块（或放入 `docs/features/S-xxxx/status.md`）：
- **Prompt 链接**：指向 `prompts/S-xxxx.md`
- **失败记录**：指向 `sessions/S-xxxx/failures.md`
- **验证命令**：如何跑、预期结果
- **风险声明**：哪些地方是“设计取舍”而不是 bug

## 5. 可观测性会被重估

很多团队过去不做可观测性，不是因为不重要，而是因为：
- 建设复杂
- 使用复杂
- 维护复杂

AI 可能改变这个等式：
- 让“复杂系统的解释与调试”变得可对话
- 让报警→定位→回滚→复盘形成更短的闭环

## 6. 放手派 vs 审查派：如何做团队级决策？

两派都可能成功，但适用边界不同。

建议用“风险分层”而不是价值观争论：
- **低风险**（原型/个人脚本）：可以更 vibe、更放手
- **中风险**（内部工具/可回滚服务）：放手 + 强可观测 + 强回滚
- **高风险**（金融/权限/安全敏感）：必须审查 + 必须证据（测试/日志/审计）

---
> 💡 结论：下一批成功产品，很可能围绕三件事展开：
> 1) Prompt/会话/失败路径的版本控制；2) 人机协作的新型 PR/Review 界面；3) 让可观测性真正“可用”。
