# Kiro 進階指南：從「會寫代碼」到「精準交付」

Kiro 不僅是一個編程插件，它是一套 **Spec-Driven (規格驅動)** 的交付系統。在 Vibe Coding 時代，Kiro 幫助我們將含糊的「Vibe」轉化為確定的「工程結果」。

## 🧠 核心洞见
> **「寫代碼」是低價值的，「交付結果」才是高價值的。**
> Kiro 的價值在於它強迫開發者在動手前先思考驗證邏輯。通過 **Spec 控制風險**、**Powers 注入知識**、**Hooks 自動化流程**，你可以將開發效率提升到一個全新的維度。

## 🛠️ 三大核心組件

### 1. Spec: 风险控制的「物理导轨」
Spec 是項目的靈魂。在 Kiro 中，我們不直接開始寫代碼，而是先寫 Specification：
- **Requirements (EARS 語法)**：使用「Easy Approach to Requirements Syntax」消除需求的歧義。
- **Acceptance Criteria (AC)**：明確「完成」的定義。
- **Tasks 分解**：將大任務拆解為具備依賴關係的離散單元。
- **驗證命令**：每個 Task 必須附帶一條可執行的驗證命令（如 `npm test` 或 `ls -R`）。

### 2. Powers: 專家知識的「動態注入」
當項目變得複雜時，全量上下文會導致模型「幻覺」和 Token 浪費。
- **按需加載**：Powers 允許你在需要時才將特定的 API 文檔、設計規範或技術選型注入 Agent 腦中。
- **上下文淨化**：只讓 Agent 知道它「現在」需要知道的事情。

### 3. Hooks: 自動化流水線的「神經反射」
Hooks 讓你的開發環境具備「自動反應」能力：
- **觸發機制**：文件保存、文件創建、Agent 任務完成等事件均可觸發。
- **自動化實踐**：
    - `onSave`: 自動運行 Linter 或單元測試。
    - `onTaskComplete`: 自動更新 `CHANGELOG.md` 或同步文檔。
    - `onInit`: 自動檢查依賴環境。

## 🚀 如何將所有流程轉化為 Kiro 自動化 Flow

要實現極致的自動化，你需要構建一個 **「意圖 -> 自動化」** 的閉環：

1.  **規範化輸入**：所有需求先經過 `requirements.md` 轉化為 EARS 格式。
2.  **配置 `STEERING.md`**：在項目根目錄定義你的全局偏好（如：禁止生成註釋、優先使用 Tailwind）。
3.  **編排 Hooks**：
    - 設置 `after_commit` Hook，讓 Agent 自動總結改動並發布到 Blog。
    - 設置 `pre_test` Hook，自動啟動 Mock 服務。
4.  **使用 Powers 擴展能力**：將常用的工具（如 `notebooklm-py`）封裝成 Power，讓 Agent 在需要時自主調用。
