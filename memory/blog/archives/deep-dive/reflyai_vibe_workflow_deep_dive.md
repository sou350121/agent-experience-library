## 一句话摘要

Refly.AI 拿到朱啸虎金沙江创投种子轮投资，核心创新是把 workflow 的每个节点变成一个配备 2-3 个工具的独立 agent，用一句话生成 workflow，成本比 Manus 低 5-7 倍，一个 Refly 节点可替代约 20 个 n8n 节点。

## 是什么 / 解决什么问题

当前 AI 工作流工具存在两极分化：一端是以 Manus 为代表的纯 Agent 形态，基于自然语言驱动但成本高、稳定性差、结果不可复现；另一端是以 n8n、Dify 为代表的传统 Workflow，强调精确性但需要编写代码维护控制逻辑，对非程序员极不友好。

Refly.AI 的解法是「Vibe Workflow」——在保留 Agent 动态性的同时，获得传统 Workflow 的可控性与稳定性。核心做法是：

- **节点即 Agent**：每个 workflow 节点本身是一个独立 agent，配备 2-3 个工具
- **沙箱执行**：给每个 agent 一台虚拟电脑，让它操作代码、拉取数据、做可视化
- **自然语言搭建**：用户只需写 prompt、选 tools，系统自动完成多步规划

创始人黄巍（前飞书 Aily 项目负责人）的判断是：传统 workflow 里复杂的 code 流程可以全部省略，变成一个节点。内部测试显示，**一个 Refly.AI 节点可以替代大约 20 个 n8n 节点的功能**。

## 技术架构拆解

### 架构本质

Refly.AI 的核心创新不是「用 AI 生成 workflow」，而是**重新定义了 workflow 节点的计算单元**：

- **传统 workflow（n8n/Dify）**：节点 = 确定性函数（API 调用/代码片段），输入输出类型固定，错误处理需显式编写
- **Refly.AI**：节点 = 概率性 Agent（LLM + 2-3 tools + 沙箱），输入输出为自然语言，错误通过重试/降级处理

这种设计把「复杂性」从用户侧转移到了模型侧：用户不需要理解 if-else 逻辑，但需要接受 70% 准确率换取 10 倍门槛降低。

### 并发与安全性

Refly.AI 的并发模型值得注意：

- **每个 Agent 节点独立沙箱**：理论上支持并行执行，但实际受限于编排引擎的顺序依赖
- **虚拟电脑隔离**：每个 agent 有独立文件系统，避免跨节点污染
- **TODO**: 未公开 rate limiting 策略，大规模并发时的 token 成本控制机制待验证

安全隐患：
- Agent 可执行任意代码，若 prompt injection 成功可能导致沙箱逃逸
- 工具调用权限边界未明确（是否能访问用户本地文件/网络）
- 多租户环境下的数据隔离策略未公开

### 核心设计决策

| 设计选择 | 理由 |
|----------|------|
| 每个节点 = 独立 Agent | 简化单点任务复杂度，让 Kimi K2 等模型也能近乎 100% 完美执行 |
| 每节点配 2-3 个工具 | 避免模型在过多工具中迷失，「选择」任务比「执行」任务简单得多 |
| 双模型分工 | 强模型（GPT-5/Claude Sonnet 4.5）负责拆任务和选工具，便宜模型（Kimi K2）负责执行 |
| 稳定编排引擎串联 | 把简单节点串起来完成复杂任务，不指望单一「大而全」的 agent |
| 放弃 100% 准确性 | 用门槛降低 10 倍的优势，换取稳定性降低 1-2 倍的代价 |

## 实用评估

### 什么场景值得用

- **自媒体内容流水线**：热点监控 → 选题生成 → 多模态内容产出（文章/播客/视频）→ 分发。Refly 自身有 3 万 Twitter 粉丝，这个场景有强体感。
- **非技术用户的流程自动化**：有 n8n/Dify 使用经验但觉得搭建复杂，或想复用他人 workflow 但不会修改的用户。
- **多模型组合任务**：想把 Claude Opus、Gemini 等不同模型串成 2-3 节点工作流，完成单一模型做不到的完整产出。
- **教育/职场/金融分析场景**：财报发布后用「巴菲特视角」写分析、监控内容做产品分析报告等偏赚钱的场景。

### 什么场景不值得用

- **需要 100% 稳定性的企业自动化**：Refly 明确放弃数字/企业自动化操作场景，节点是 agent 就达不到 n8n 的准确度。
- **精确计算/数据库操作**：涉及金融交易、库存管理等对准确率要求极高的场景。
- **超大规模并发任务**：目前产品还在早期，工程复杂度虽高但未经过大规模压力验证。
- **纯聊天/问答场景**：如果只是简单问答，直接用 ChatGPT/Claude 即可，无需 workflow。

## 理论好用 vs 实际好用

### 理论上好用的场景

- 自媒体内容流水线（热点 → 选题 → 多模态产出 → 分发）
- 非技术用户复用他人 workflow 模板
- 多模型组合任务（Claude + Gemini 串联）

### 实际上可能的陷阱

| 预期 | 现实 |
|------|------|
| 一句话生成完美 workflow | 需多次迭代调整 prompt 和 tools 选择 |
| 一个节点替代 20 个 n8n 节点 | 仅在简单场景成立，复杂逻辑仍需多节点拆分 |
| token 消耗降低 5-7 倍 | 仅限执行层，规划层用强模型成本不低 |
| 模板复用 token 降至 10% | 需 workflow 完全匹配，微调后成本回升 |
| 70% 准确率足够用 | 对内容创作可能够用，但对数据提取/分析可能不够 |

### 隐性成本

- **调试成本**：Agent 节点出错时，定位问题比传统 workflow 更难（需分析 prompt/tool 选择/模型输出）
- **版本锁定**：workflow 依赖特定模型版本，模型更新后行为可能变化
- **模板维护**：发布到社区的模板需持续更新以适配模型/工具变化

## Claude Code 盲点

如果你用 Claude Code 搭建类似 Refly.AI 的系统，以下陷阱需注意：

❌ **错误做法**：让单个 Claude 实例同时负责任务拆解 + 工具选择 + 执行
- 原因：上下文爆炸，工具选择准确率随工具数量增加而下降
- 正确：拆分为规划模型（强）+ 执行模型（便宜），每节点限 2-3 tools

❌ **错误做法**：用自然语言直接串联所有步骤，不做结构化编排
- 原因：长链条中错误累积，中间步骤失败无法优雅降级
- 正确：用稳定编排引擎管理节点依赖，支持重试/跳过/人工接管

❌ **错误做法**：假设用户 prompt 足够清晰，不做意图校验
- 原因：用户表达模糊时，生成的 workflow 可能完全偏离预期
- 正确：增加「意图确认」步骤，用示例 workflow 让用户确认后再执行

❌ **错误做法**：把所有工具都暴露给规划模型
- 原因：工具过多导致选择 paralysis，且增加 token 成本
- 正确：按场景分组工具，规划时先用粗粒度分类，再在节点内细选
