# 2025 LLM 现状：进展、问题、以及对 2026 的工程预测（Raschka 摘要提纯）

> 来源：`https://magazine.sebastianraschka.com/p/state-of-llms-2025`
> 
> 目标：把“年度盘点”提纯为 **可执行的工程决策清单**，用于 Agent / Coding Agent / RAG 系统选型与治理。

---

## 1. 2025 的主旋律（工程视角）

### 1.1 推理成为主线：Reasoning / RLVR / GRPO
- **现象**：推理模型在数学等任务上快速逼近顶尖水平，且进展比许多人预期更早出现。
- **工程含义**：
  - 更强的“可验证推理”会推动产品形态从“会说”走向“会做（并能自证）”。
  - 训练/对齐策略（如 RLVR/GRPO）对能力增长的贡献提升，模型迭代会更快、更非线性。

### 1.2 推理时扩展 + 工具使用：系统层成为性能放大器
- **现象**：inference-time scaling 与 tool use 变成核心方向。
- **工程含义**：
  - “换更强模型”不再是唯一杠杆。
  - **调度、工具链、上下文投喂、执行闭环（ReAct）** 会直接决定最终效果。

延伸阅读（可执行清单）：
- `docs/capabilities/post-scaling-research-age-playbook.mdx`

### 1.3 Benchmaxxing：基准刷榜与真实价值脱钩风险
- **工程含义**：
  - 评估体系必须回到：真实任务、真实成本、真实失败模式。

### 1.4 开放权重生态迁移：更多选择、更碎片的现实
- **现象**：开源/开放权重社区偏好发生变化（Raschka 指出 Qwen 超越 Llama 的趋势），同时更多竞品涌现。
- **工程含义**：
  - 需要 **模型替换成本最小化** 的架构（接口抽象、评测回归、prompt/skill 版本控制）。

### 1.5 MCP 更快成为 agent 工具/数据访问标准
- **现象**：Raschka 提到 MCP 加入 Linux Foundation 后，在 agent-style 系统中快速成为标准。
- **工程含义**：
  - 工具能力标准化会降低“生态碎片化”的摩擦成本。
  - 架构上应把 MCP 当作“默认工具总线”，而不是临时胶水。

## 2. 2026 的工程预测：你应该怎么准备

### 2.1 预测：开放权重会逐步具备本地工具使用与更强 agentic 能力
- **行动项**：
  - 把“工具访问层”做成可插拔（MCP/Adapter），避免绑定单一供应商。

### 2.2 预测：经典 RAG 会逐渐退出“默认解法”
- **Raschka 观点**：不再对每个文档问题都检索，而更依赖长上下文与更好的小模型。
- **行动项**：
  - 把 RAG 从“默认必走路径”改为“按需启用的策略”。
  - 建立策略路由：何时检索、检索多少、何时只用 long-context。

### 2.3 预测：更多进展来自 tooling 与 inference-time scaling
- **行动项**：
  - 投资于：工具质量、观测体系、失败路径收敛、可复现评测与回归。

## 3. 可执行 Checklist（落地到你的 Agent 系统）

### 3.1 评估（避免 benchmaxxing 陷阱）
- [ ] 用 3-5 个真实任务做评估（而不是只看榜单）
- [ ] 记录成本：延迟、token、失败率、人工返工率
- [ ] 输出失败模式：哪些任务经常“看似对、实则错”

### 3.2 架构（系统层放大能力）
- [ ] ReAct 闭环：工具调用必须可观察、可回放
- [ ] 工具总线：优先 MCP（或至少保持兼容）
- [ ] 上下文治理：@ 精准投喂 + 子代理隔离（避免污染主上下文）

### 3.3 治理（适应模型快速迭代）
- [ ] 建立 Prompt/Skill 版本控制（可回滚、可对比）
- [ ] 建立评测回归（每次换模型/调策略都跑）
- [ ] 把确定逻辑抽成 Skill/Script（幂等安全墙），缩小非幂等区域

### 3.4 RAG 策略（从默认变成按需）
- [ ] 为每类问题定义：是否检索、检索源、检索 top-k、证据引用格式
- [ ] 对长上下文直接回答的场景，仍要求可验证证据（引用输入片段）
