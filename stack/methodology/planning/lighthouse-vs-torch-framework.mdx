# 灯塔 vs 火把：Agent 选型决策框架

在开发 Agent 时，选择“大脑”不仅仅是看榜单分数，更是一场关于**分配权、隐私与稳定性**的权衡。

## 选型矩阵

| 需求场景 | 推荐光源 | 代表模型 | 理由 |
| :--- | :--- | :--- | :--- |
| **架构设计/长程规划** | 🗼 灯塔 | GPT-5.2 / Claude Opus 4.5 | 在超长上下文和多文件逻辑一致性上具备断层优势 |
| **本地代码补全/重构** | 🔥 火把 | DeepSeek-V3 / Qwen-2.5 | 响应极快，AI 已“磨平”了 TypeScript 等繁琐语法 |
| **私有知识库 (RAG)** | 🔥 火把 | Local Models (via Ollama) | 数据主权完整，无 API 成本负担，适合处理一生私密数据 |
| **跨领域前沿探索** | 🗼 灯塔 | GPT-5.2 Thinking / Gemini 3 Pro | 触碰认知边界，语感进化，更像人类的深度逻辑推理 |

## 双轨工作流 (Hybrid Workflow)

我们推荐在复杂的 Agent 系统中采用以下工作流：

1. **灯塔规划 (Planning)**：由 SOTA 模型根据需求生成 `implementation-plan.md`。
2. **火把执行 (Execution)**：Agent 拿着计划，在本地由开源模型执行具体的编码或数据处理任务。
3. **灯塔审计 (Review)**：关键步骤交回给强模型进行安全与质量把关。

## 核心思考
> “灯塔指引远方，火把守住脚下。”

对于 Agent 使用者来说，最高级的境界是：**当网络断开或 API 调价时，你手里依然握着一束不必向任何人借的光。**
