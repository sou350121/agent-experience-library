## 1) 本周信号（Many）

### 1.1 推理引擎不再只拼 tokens/s，而是拼“系统层能力”
- **vLLM 语义路由 v0.1** 把“模型选择 + 安全过滤（脱狱/PII）+ 语义缓存 + 幻觉门控”抽成了一个独立层。
- 这意味着推理层正在往上长出 **“决策/治理/策略”** 的能力，而不仅是执行。

参考：
- [vLLM 巨大里程碑：语义路由 v0.1 与“小号 vLLM”生态](./2026-01-06-vllm-milestone-semantic-routing.md)
- [vLLM 语义路由深度解析](../stack/frameworks/vllm-semantic-routing-deep-dive.mdx)

### 1.2 “小号 vLLM”生态涌现：推理内核开始可教学、可定制
- `nano-vllm / MinivLLM / tiny-llm` 的意义不是替代 vLLM 本体，而是把关键抽象缩小到“可读、可改、可嵌入”的规模。
- 对个人/小团队来说，这类项目最好的用途是：**做架构理解与轻量实验的底座**。

### 1.3 Agent 生产化的门槛：从“能写代码”变成“可治理、可审计”
- 你需要的不只是一个更聪明的模型，而是一套能把不确定性关起来的工程结构。

## 2) 这周真正发生的变化（One）

把本周所有噪音压缩成一句话：

**“LLM 系统开始从‘模型中心’迁移到‘路由/治理中心’。”**

换句话说：
- 未来让你系统变强的，不一定是换更大的模型；
- 而是你有没有一个足够强的 **路由层**，能在成本/安全/质量之间做稳定决策。

---

## 3) 生存公式（Formula）

如果你只记住一个公式：

```text
SystemReliability ∝ (IdempotentGovernance + RoutingPolicy) / ModelRandomness

IdempotentGovernance = Skills/Script/可审计工具链
RoutingPolicy        = 语义路由(选模型 + 安全 + 缓存 + 质量门控)
ModelRandomness      = prompt 非幂等 + 输出不稳定
```

## 4) 你这周该做什么（可执行清单）

### 4.1 给你的 Agent 加一层“路由前置”
- [ ] 先把请求分成 3 类：**低风险/高频**、**高风险**、**高成本**
- [ ] 为每类定义默认去向：便宜模型/高能力模型/需要工具的模型
- [ ] 把 PII/脱狱拦截写进“墙内规则”（别写在 prompt 里）

### 4.2 把缓存从“字符串缓存”升级为“语义缓存”
- [ ] 先挑 1 个场景：FAQ、代码 review 建议、重复 PRD 生成
- [ ] 定义 cache key：输入语义 + 版本号（模型版本/策略版本）
- [ ] 明确命中后的行为：直接返回 / 作为候选 / 触发二次验证

### 4.3 如果你要本地部署：先用“小号 vLLM”做教学与验证
- [ ] 目标不是跑 SOTA，而是跑通：调度/KV cache/并发/错误处理
- [ ] 等你对执行链路足够清楚，再切到 vLLM 本体做性能化

### 4.4 把“幂等安全墙”再加厚一厘米
- [ ] 把高风险动作收口到脚本：写文件/删文件/发布/外网请求
- [ ] Agent 只负责“决策”，脚本负责“执行”（可审计、可回放）

## 5) 本周 Prompt（荒野生存版）

```text
目标：把我的 LLM/Agent 系统从“模型中心”升级为“路由/治理中心”。

请输出：
1) 一个三层路由策略（low-risk/high-risk/high-cost）
2) 每层的默认模型/提供商选择规则
3) 安全门控（PII/脱狱）放在哪一层、如何回退
4) 语义缓存的 cache key 设计（含版本化）
5) 失败模式与熔断策略

约束：
- 所有高风险动作必须进入“幂等安全墙”（脚本/工具）
- 不要写空泛建议，要给可执行 checklist
```

- **语义路由是否会成为推理层标配**（像 API Gateway 一样必备）
